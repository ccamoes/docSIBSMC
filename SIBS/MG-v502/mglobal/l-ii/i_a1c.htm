CARACTERIZAÇÃO DO BIN


Nesta caracterização, o Emissor indica os parâmetros pretendidos para cada tipo de cartão.

Aquando do início de funcionamento de um novo BIN, ou sempre que se pretenda alterar uma das parametrizações em uso, o Emissor define os parâmetros do BIN através da funcionalidade de Pedido de Caracterização de Emissores disponibilizada no Terminal de Serviços SIBS. Para situações de excepção em que tal não seja possível, o Emissor deve preencher e enviar à SIBS este formulário.

O Emissor indica o ambiente em que pretende realizar parametrizações por escolha de uma das opções para o “Tipo de Caracterização”:

  • Pré-Produção - as parametrizações indicadas referem-se exclusivamente ao ambiente de Pré-Produção. O Emissor indica posteriormente se pretende anular estas parametrizações ou integrá-las em Produção;
  • Produção (e em Pré-Produção) - as parametrizações indicadas são efectuadas em paralelo nos ambientes de Pré-Produção e de Produção.

 


1. CÓDIGO DO EMISSOR

Indique o número de identificação do Emissor no Sistema Multibanco.


2. BIN

Indique os oito dígitos correspondentes ao BIN e respectiva extensão do produto-cartão.


3. CLASSIFICAÇÃO DO PRODUTO CARTÃO

3.1.  
DESIGNAÇÃO COMERCIAL

Indique a designação comercial que identifica o produto-cartão.

   

3.2.  
TIPO DE CONTA ASSOCIADA

A classificação do produto-cartão é usada unicamente para fins estatísticos, de acordo com as orientações do Banco de Portugal.

   

3.3.
FAMÍLIA DE CARTÃO

Indique o sistema ou sistemas de pagamento a que pertence o cartão, por indicação da família de cartão correspondente.

Possibilita-se que emissores bancários nacionais emitam cartões sem a marca Multibanco. Para cartões que possuam em simultâneo a marca Multibanco e uma outra marca, é necessário indicar a família de cartão correspondente à combinação de ambas as marcas. 

A escolha de uma família de cartão que não inclua a marca Multibanco condiciona as operações disponibilizadas na Rede Multibanco.

Para mais informações sobre as operações disponíveis no âmbito de cada marca, consulte o capítulo A.6.
   

3.4.
CÓDIGOS DE SERVIÇO (SERVICE CODE)

Indique se pretende utilizar um Service Code específico para os cartões deste BIN diferente dos valores habituais para o produto associado.

   

3.5.
CHIP NÃO-EMV

Indique se o produto-cartão possui ou não chip não-EMV.
Em caso afirmativo, o Emissor deve indicar que aplicações se encontram disponíveis:

  • Só a aplicação de Porta Moedas MB (cartão combinado), ou
  • A anterior mais uma segunda aplicação com outras características (ex.: controlo de acessos).
4. TENTATIVAS DE PIN

Indique qual deve ser o procedimento a executar quando o cliente erra o código secreto pela terceira vez consecutiva num Caixa Automático MB:

  • Capturar o cartão, podendo o cliente reavê-lo após o fecho contabilístico do CA ou
  • Expulsar o cartão, deixando este de ser utilizável para operações de uso electrónico.

5. FORMATO DO NÚMERO DE CARTÃO NOS RECIBOS

O Emissor deve escolher a identificação que melhor se adeque ao cartão. Normalmente em cartões Visa utiliza-se o formato 'BIN e número de cartão' e nos MB ou Proprietários, o formato 'Código de Emissor e número de cartão'.


6. BIN PARA CARTÕES TEMPORÁRIOS DO SERVIÇO MBNet

O Emissor deve indicar o BIN utilizado para a geração de cartões temporários associado ao serviço MBNet.


7. DESTINO DE FICHEIROS (File Transfer) E PROCESSAMENTOS OPCIONAIS

7.1.  
ENDEREÇO PARA ENVIO DE FICHEIROS  

Indique o endereço a utilizar para envio dos ficheiros seguintes, quando se pretenda agregar a informação por BIN:

  • Destinos (DST5) 
  • Captura e Lista Negra (CLN5)

A indicação deste endereço de File Transfer é opcional. 

A agregação de informação relativa à utilização de cartões é agregada pelo destino de File Transfer aplicável. Quando não for indicado nenhum endereço neste ponto da caracterização, os registos relativos às movimentações de cartões deste BIN são incluídos nos ficheiros enviados para o endereço indicado na Caracterização do CPD ao qual pertencem os cartões.

 

7.2.
INDICADORES DE PROCESSAMENTOS ADICIONAIS
  • Ficheiro Detalhes: indique se o Emissor pretende recepcionar o ficheiro DET5. Se SIM, este ficheiro é enviado para o endereço de File Transfer do BIN ou, se este não estiver definido, para o endereço de File Transfer do CPD.

  • Ficheiro Pagamentos de Baixo Valor: indique se o Emissor pretende recepcionar o ficheiro PBV5. Se SIM, este ficheiro é enviado para o endereço de File Transfer do BIN ou, se este não estiver definido, para o endereço de File Transfer do CPD.

  • Ficheiro Autorizações AUT5: indique se o Emissor pretende recepcionar o ficheiro AUT5. Se SIM, pode opcionalmente indicar um endereço específico utilizado exclusivamente para o envio deste ficheiro. A não definição de um endereço específico implica o envio do AUT5 para o endereço de File Transfer do BIN ou, se este não estiver definido, para o endereço de File Transfer do CPD. 
8. OBSCURECIMENTO DE CONTA NOS TALÕES DE CA

Indique se pretende que a conta associada ao cartão não seja apresentada nos talões entregues ao cliente, relativos a operações realizadas em Caixas Automáticos da Rede Multibanco com impressão de talão.


9. LIMITES

9.1.  
LIMITE EM CENÁRIO DE DEGRADAÇÃO EM TPA 

No funcionamento do cenário Real-Time não existe controlo de limite máximo para os montantes de operações de compra, de pagamento de serviços/compras e transferência interbancária aceites na Rede Multibanco. Deste modo, o Emissor deve definir um eventual limite para estas operações-cliente (por razões de segurança).

Porém, no caso de estar em funcionamento o cenário de Saldo de Conta ou de Cartão, a SIBS efectua esse controlo com base num montante máximo diário definido pelo Emissor, com excepção para os pagamentos de serviços/compras.
Mesmo que a operação solicitada fosse aceitável pelo Saldo de Cartão ou de Conta, presente na SIBS, o sistema MB verifica se o total de compras feitas pelo cartão nesse dia adicionado ao que se pretende efectuar, não ultrapassa esse montante.

   

9.2.
LIMITE EM SERVIÇO REDUZIDO EM TPA

Em caso de falha de comunicação com o Host da SIBS, um TPA pode ficar a funcionar através de um equipamento Front-End (FEP-POS), que controla uma grande área de TPAs. O Emissor pode indicar qual o montante máximo diário por cartão, que o FEP pode aceitar.

Para mais informações, consultar o capítulo Serviço Reduzido.

   

9.3.
LIMITE EM PAGAMENTOS DE BAIXO VALOR

Existem alguns serviços Multibanco para os quais não é necessária a introdução de código secreto: são as operações de baixo valor. Pretende-se que o Emissor defina qual a importância que a SIBS deve usar para este objectivo.


10. OPERAÇÕES NO ESTRANGEIRO

10.1.  
CENÁRIO DE LIMITE DE AUTORIZAÇÃO

Esta alínea deve ser preenchida pelos emissores que pretenderem que a SIBS aplique um cenário de degradação nas operações que são feitas no estrangeiro, em sistemas de pagamento cujas mensagens não são financeiras (ex.: Visa, MasterCard, AMEX).
Para mais informações, consulte o capítulo Cenário de Limite de Autorização.

   

10.6.
ACÇÃO PARA ERRO OU TIME OUT  NA RESPOSTA EM REAL-TIME

Indique o procedimento da SIBS junto do sistema de pagamento internacional quando não há uma resposta em tempo real do Emissor.
Para mais informações, consultar o capítulo Cenário de Limite de Autorização.


11. CARACTERIZAÇÃO DE OUTRAS VERTENTES

Esta alínea deve ser preenchida no caso do cartão, possuir uma outra vertente para além do Multibanco (por exemplo, pertencer a um sistema de pagamento internacional, com condições especiais na rede de comerciantes gerida pelo seu representante em Portugal (ex: Unicre para cartões Visa e MasterCard).


11.1.  
AUTENTICAÇÃO DAS OPERAÇÕES

Assinale qual a modalidade que o cliente pode utilizar em TPAs com atendimento. No caso de optar pela alternativa de uso de código ou assinatura, à escolha, a SIBS valida o código secreto apenas se o cliente o introduzir. Neste caso, a operação só pode ser aceite se o código estiver correcto. No caso do código não ter sido introduzido, o cliente deve assinar o talão da compra.

   

11.2.  
REGIME DE PRICING

Os comissionamentos relativos a transacções realizadas na rede do Representante em Portugal podem estar indexados a valores previamente definidos. Quando existe um acordo entre o Emissor e o Representante para a aplicação de um regime de comissionamento específico (regime de pricing), indique o código atribuído a esse regime de comissionamento. 

   

11.3.  
PRODUTO FINANCEIRO (ACORDO) ASSOCIADO AO BIN

A aceitação de cartões em TPA numa vertente diferente da vertente Multibanco é efectuada no âmbito de um acordo específico, carregado nos terminais e habitualmente designado por Produto Financeiro.

Indique neste ponto o Produto Financeiro (principal) associado ao BIN.

No caso específico de cartões representados pela Unicre (cartões com uma marca da Visa ou da MasterCard), o Produto Financeiro (principal) a utilizar está indexado ao Tipo de Conta associada ao cartão:

  • se o Tipo de Conta indica uma vertente de Débito: o Produto Financeiro (principal) a indicar é obrigatoriamente "00347-UNICRE DB";

  • se o Tipo de Conta indica uma vertente de Crédito: o Produto Financeiro (principal) a indicar é obrigatoriamente "00346-UNICRE CR".

Exclusivamente nas situações em que o Produto Financeiro (principal) indicado seja "00347-UNICRE DB", deve ser indicado como Produto Financeiro (secundário) o "00346-UNICRE CR", para garantir a normal aceitação dos cartões.

   

11.4.
REPRESENTANTE DO PRODUTO-CARTÃO
  1. Indique quem é o representante dos cartões, isto é, a entidade que negoceia com o comerciante as condições contratuais para a aceitação da marca (que não MB) associada ao cartão.
  2. Indique qual é o acordo de partilha da comissão (paga pelo comerciante nas operações feitas com cartões do Emissor) que o Emissor tem com o representante do cartão. Esta partilha também se aplica às comissões pagas pelos comerciantes a cartões Multibanco, em alguns acordos entre o representante e o comerciante. Por isso, solicita-se o preenchimento, também, para o caso do cartão ser MB ou Proprietário.
  3. Indique se nas transacções realizadas em gasolineiras por cartões deste BIN deve ser aplicada a taxa gasolineira (€ 0,50).
  4. Indique qual o encaminhamento que deve ser dado às operações recebidas de TPAs pertencentes à rede do representante, sendo um cartão desse âmbito:
    • Emissor, em Real-Time: significa que este processa as referidas operações tal como as outras compras dos cartões Multibanco, pressupondo que o Emissor tem cenário Real-Time (Emissor autónomo);
    • Representante, em Real-Time: significa que a SIBS encaminha essas operações para o representante (ex.: Unicre), validando previamente a existência do cartão no seu sistema e a autenticidade do PIN. A Compensação da operação é feita através do Representante (Emissor não autónomo);
    • SIBS, em saldo de Conta/Cartão/Crédito Disponível: significa que o Emissor pretende que a SIBS trate estas operações utilizando o cenários de Saldo de Conta, Saldo de Cartão ou Saldo de Crédito Disponível. A opção escolhida para o cenário principal determina o destinatário da compensação das transacções (envio directo ao Emissor ou ao Representante).

Se o centro de autorizações for o Emissor em Real-Time, este deve indicar qual o cenário de degradação das operações que não são decididas pelo Emissor.

 
11.5. ENCAMINHAMENTO AUTOMÁTICO DE INFORMAÇÃO SOBRE LISTA NEGRA

No caso de inserções em Lista Negra ou Lista Cinzenta não originadas no Emissor, este deve indicar quais são os representantes (nacional e/ou internacional) que a SIBS deve informar automaticamente:

  • Inserções detectadas na rede SIBS (Lista Negra da rede MB);
  • Inserções recebidas como 'lista negra urgente';
  • Inserções detectadas na Rede SIBS (Lista Cinzenta da Rede MB).

12. CARACTERIZAÇÃO DE ELEMENTOS EMV

Esta alínea é de preenchimento obrigatório para possibilitar a emissão de cartões EMV, com a excepção de alguns pontos em que tal é expressamente indicado.
Os elementos parametrizados através desta alínea serão utilizados quando os cartões sejam de tecnologia EMV.


12.1.  
ÂMBITO DE UTILIZAÇÃO

Neste ponto o emissor pode restringir o âmbito de utilização geográfico e os serviços permitidos à aplicação EMV.
Assinale os serviços a disponibilizar.

   

12.2.
PARÂMETROS DE RISCO: TRANSACÇÕES OFFLINE

Quando a aplicação que está a ser parametrizada tem correspondência com um produto de um Sistema de Pagamentos Internacional habitualmente visto como de débito (ex: Visa Electron e Maestro), sugere-se o não preenchimento do ponto a. (segunda moeda) e que os parâmetros de gestão de risco para operações no estrangeiro, no ponto c. (Transacções Internacionais), sejam preenchidos com "0" (zeros). Desta forma, ainda que a nível doméstico o emissor disponibilize funcionalidades offline para estes cartões, garante-se que as transacções realizadas são 100% autorizadas, minimizando-se assim o risco associado a transacções no estrangeiro com validação de PIN Offline.

  1. Código da segunda moeda da aplicação

Este ponto só deve ser preenchido se o emissor desejar posicionar um código de uma segunda moeda para a Aplicação EMV em causa.
A existência deste código permite uma validação adicional dos montantes de transacções offline realizados na segunda moeda.
O código de segunda moeda é parametrizado na emissão do cartão e não é actualizável posteriormente.
Para as restantes transacções offline que não sejam realizadas na moeda principal da aplicação ou na segunda moeda, apenas são aplicáveis parâmetros de risco baseados em número de transacções (não são aplicáveis outros parâmetros de risco em valor).

  • Código da segunda moeda:
    O emissor deve indicar uma representação de 3 caracteres numéricos, de acordo com a norma ISO 4217.
  • Descritivo:
    Texto para confirmação do código da segunda moeda. 
  • Factor de conversão:
    Valor decimal utilizado para conversão de valores da segunda moeda para a moeda principal da Aplicação EMV (o euro).
    O factor de conversão é actualizado apenas por indicação do emissor. Pode ser actualizado em qualquer momento da vida útil do cartão (via ficheiro EGCC).
    Trata-se de um valor aproximado, utilizado apenas para a validação do Montante Máximo Acumulado. Recomenda-se a sua actualização apenas quando se considere que sofreu uma variação que o justifica.
  1. Limites sobre os montantes das transacções
  • Quantia Máxima Acumulada permitida para Transacções Offline, mesmo que o terminal tenha a possibilidade de efectuar a transacção Online:
  • MasterCard: este limite é aplicável a todas as transacções realizadas na moeda principal da Aplicação ou na segunda moeda, quando esta exista.
  • Visa: este Sistema de Pagamentos considera a existência de dois elementos distintos, estabelecendo um "limite na moeda principal da aplicação" e um "limite dual currency", quando exista uma segunda moeda. Na caracterização de aplicações Visa, os referidos dois elementos assumem o mesmo valor (parametrizado neste campo do Padrão EMV).
  • Quando ultrapassado este limite, tenta-se decidir a transacção online. Se tal não for possível, a transacção pode ser realizada offline.
  • Quantia Máxima Acumulada permitida para Transacções Offline a partir da qual a transacção tem obrigatoriamente que ser decidida online:
  • Este limite é aplicável a todas as transacções realizadas na moeda principal da Aplicação ou na segunda moeda, quando esta exista.
  • Quando ultrapassado este limite, a transacção tem que ir online. A impossibilidade de decisão online implica a recusa da transacção.
  • Este elemento existe apenas se o cartão suportar a norma VIS 1.4.0 (Visa) ou a norma MChip/4 (MasterCard), ou versões superiores.
  1. Transacções Internacionais

Preenchimento livre pelo emissor (existem contudo recomendações dos Sistemas de Pagamento para o preenchimento destes valores).

  1. Limites globais

Preenchimento livre pelo emissor (existem contudo recomendações dos Sistemas de Pagamento para o preenchimento destes valores).

 
12.3. LISTA DE MÉTODOS DE VERIFICAÇÃO DO PORTADOR (CVM List)
  1. Montante abaixo do qual se permite o PIN Offline

Esta condição corresponde a um método adicional de Verificação do Portador.
Como elemento adicional de gestão de risco, o emissor pode parametrizar opcionalmente um valor abaixo do qual pode ser utilizado o método de verificação "PIN Offline", quando este exista na lista de métodos do cartão; é aplicável para cartões DDA e quando o terminal suporta PIN Offline.

A aplicabilidade e restrições impostas por este método dependem das capacidade do Terminal:

  • Quando o terminal suporta PIN Offline e o montante está abaixo do indicado, é tentado o PIN Offline como primeiro método. Se não for validado o PIN Offline, utiliza-se o Método de Verificação seguinte.
  • Quando o terminal não suporta o PIN Offline, ignora-se o limite apresentado neste ponto e utiliza-se o próximo Método de Verificação do Portador (incluído na lista de CVMs) que seja aplicável.
  1. Listas de CVM alternativas

O emissor pode escolher apenas uma das 3 hipóteses alternativas indicadas neste ponto.

As 3 hipóteses apresentadas têm em consideração diversos factores:

  • As recomendações dos Sistemas de Pagamento Internacionais para os diversos produtos existentes.
  • O comportamento pretendido para o funcionamento dos cartões on-us (possibilitar a validação por assinatura para sectores determinados, como por exemplo os restaurantes; para os restantes sectores, considerar o PIN Obrigatório).
  • A validação adicional introduzida no ponto 12.3.a).
  • A tecnologia do chip EMV do cartão: assume-se que é desaconselhável a utilização do PIN Offline em claro, pelo que o PIN Offline não é um dos métodos suportados quando os cartões possuem uma tecnologia SDA (Static Data Authentication).

Sobre as recomendações dos Sistemas de Pagamento Internacionais, recorda-se que:

  • A "Assinatura" deve existir nos produtos com vertente de crédito (ex: Visa e MasterCard).
  • Se for pretendida a utilização dos cartões em causa em terminais que apenas suportam PIN Online, este tem que ser incluído na lista de métodos.
  • Os Visa Electron podem ter PIN Offline (opcional).
  • Se existir o método "No-CVM", este deve ser o último da lista de métodos de verificação.
  • O método "No-CVM" deve existir para os produtos com vertente de crédito (ex: Visa e MasterCard).
  • O método "No-CVM" pode ser incluído nos produtos da Visa normalmente associados a produtos de débito (Visa Electron).
  • O método "No-CVM" não pode ser incluído nos produtos da MasterCard normalmente associados a produtos de débito (Maestro).

O detalhe das alternativas de Lista de Métodos de Verificação do Portador são apresentadas de seguida. Indicam-se também as regras aplicáveis relativamente a cada um dos métodos de verificação suportados:

12.4
FALLBACK PARA PISTA MAGNÉTICA 

Em qualquer transacção com uma aplicação de pagamento, no diálogo com o cartão o terminal efectua uma primeira validação do conteúdo da Pista 2. Se esta indicar que o cartão utilizado tem chip EMV, o terminal tenta ler os elementos do chip.
Prevendo as situações em que não é possível efectuar uma correcta leitura dos dados contidos no chip, possibilita-se que o emissor especifique se pretende que a decisão da transacção seja efectuada com base nos elementos da pista magnética (Fallback).

  1. As transacções podem ser decididas com recurso a Fallback?

O emissor deve indicar expressamente se pretende autorizar a possibilidade de Fallback para os seus cartões.

Elementos parametrizáveis:

  • Número máximo de operações consecutivas permitidas
    Este campo só deve ser preenchido se o emissor indicou que pretende possibilitar o Fallback. 
    Limita o número de transacções com recurso a Fallback consecutivas. Quando este limite é atingido, qualquer transacção posterior só pode ser realizada com base nos elementos do chip, não se permitindo que o cartão continue a efectuar operações exclusivamente sobre os dados da pista magnética.
  • Valor máximo por operação
    Este campo só deve ser preenchido se o emissor indicou que pretende possibilitar o Fallback. 
    Montante máximo para cada transacção com recurso a Fallback, considerada individualmente.

13. CARACTERIZAÇÃO DE LINHA DE CRÉDITO (PRESTAÇÕES)

Este ponto possibilita a parametrização de um número de ocorrências (máximo de 9) para conjuntos de elementos necessários à operacionalidade da funcionalidade "Linha de Crédito". Se o Emissor pretender disponibilizar esta funcionalidade em cartões EMV emitidos para o BIN em causa, tem obrigatoriamente que preencher pelo menos uma ocorrência (linha da tabela).

A cada linha da tabela corresponde uma identificação (ID), a indicar no momento da emissão de novos cartões (via ficheiro EECB) ou em actualizações após a emissão (via EGCC).


14. TEXTOS PARA TERMINAIS (ESPECÍFICOS DO BIN)
